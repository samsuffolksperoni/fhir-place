# Audit Model

How the workbench persists agent runs, and how that shape maps to
FHIR's audit-log resources. Phase A **never writes** `AuditEvent` or
`Provenance` resources back to the FHIR server â€” write-back is
icebox. The vocabulary is borrowed because it's the right
vocabulary, not because we're shipping a FHIR audit-log producer.

## What is persisted

Three tables, all in the same SQLite file as
`data_connection` and `agent_session`:

| Table | One row per | FK to | Lifetime |
| --- | --- | --- | --- |
| `agent_answer` | `runPatientSummary` invocation | `agent_session(id)` (cascade) | Until session is deleted |
| `tool_call` | tool execution (agent or debug runner) | `agent_session(id)` (cascade), `agent_answer(id)` (set null) | Until session is deleted |
| `evidence_claim` | `EvidenceBackedClaim` in a persisted answer | `agent_answer(id)` (cascade) | Until answer is deleted |

Schema details: [`db/migrations/0003_audit.sql`](../db/migrations/0003_audit.sql)
and the Drizzle mirror in [`db/schema.ts`](../db/schema.ts).

### `agent_answer`

| Column | Type | What it is |
| --- | --- | --- |
| `id` | text PK | Pre-generated by the answers route. Stamped onto every `tool_call.answer_id` for the run. |
| `session_id` | FK to `agent_session.id` | The session under which this run happened. |
| `prompt` | text | The user-supplied or default prompt. |
| `prompt_version` | text | `patient-summary@v1` today; bump in `prompts.ts` to evolve. |
| `provider` | text | `anthropic`. |
| `model` | text | Resolved at request time (`WORKBENCH_AGENT_MODEL` or default). |
| `fallback` | 0/1 | True iff the orchestrator built a partial-answer fallback (max-turn, end-turn-without-finalize, or two consecutive validation failures). |
| `turns` | integer | Number of model calls made. |
| `answer_json` | text | The validated `AgentAnswer` body, verbatim. The orchestrator never persists an answer that didn't pass schema validation. |
| `final_issues_json` | text? | Zod issues, when the orchestrator fell back to a partial answer due to validation failure. |
| `created_at` | text | ISO timestamp. |

### `tool_call`

| Column | Type | What it is |
| --- | --- | --- |
| `id` | text PK | Generated by the audit store. |
| `session_id` | FK to `agent_session.id` | The session under which the call happened. |
| `answer_id` | FK to `agent_answer.id` (nullable) | Set during agent runs; NULL for debug-runner calls. |
| `connection_id` / `patient_id` | text | Denormalised snapshot â€” replay-safe even if the session row is later mutated. |
| `tool` / `tool_version` | text | E.g. `searchConditionsForPatient` / `1`. |
| `input_json` | text | The registry-accepted input. **No auth tokens** â€” tools never receive the connection's `authToken`. |
| `envelope_json` | text | The full normalised `ToolEnvelope` (PR 4) including upstream `Bundle` `searchset` payloads on success and `OperationOutcome` bodies on error. |
| `ok` | 0/1 | Mirror of `envelope.ok` for cheap indexing. |
| `reason` | text? | The `ToolErrorReason` for `ok = 0`. |
| `result_count` / `truncated` | integer? | From the envelope. |
| `duration_ms` | integer | From the envelope. |
| `resource_ids_json` | text? | Flat `["Type/id", â€¦]` array extracted from successful envelopes. Same shape as the renderer's evidence chips. |
| `started_at` / `completed_at` | text | ISO timestamps. |

### `evidence_claim`

| Column | Type | What it is |
| --- | --- | --- |
| `id` | text PK | Generated by the audit store. |
| `answer_id` | FK to `agent_answer.id` (cascade) | Owning answer. |
| `claim_id` | text | The `AgentAnswer` claim's `id`. |
| `text` | text | The claim's `text`. |
| `evidence_refs_json` | text | The `ResourceReference[]` as JSON. The truth is still `agent_answer.answer_json`; this is denormalised data so audit queries don't have to JSON-parse the answer. |

## FHIR-adjacent mapping

If we ever did write back, this is how the rows would map to FHIR
audit-log resources. None of this is implemented today.

### `agent_answer` â†’ `Provenance`

| `Provenance` element | This table |
| --- | --- |
| `target.subject` | `agent_session.patient_id` (joined via `session_id`) |
| `recorded` | `agent_answer.created_at` |
| `agent.who.display` | `provider`/`model` |
| `agent.role` | `"author"` (the agent generated the answer) |
| `entity[*].what` | The cited `<Type>/<id>` references in `evidence_claim.evidence_refs_json` |
| `reason` | `prompt` |
| `signature` | n/a â€” out of scope for Phase A |

### `tool_call` â†’ `AuditEvent`

| `AuditEvent` element | This table |
| --- | --- |
| `category.text` | `"agent_tool_call"` |
| `code.text` | `tool@toolVersion` |
| `recorded` | `started_at` |
| `outcome.code` | `0` for `ok = 1`, `8` (serious failure) for `ok = 0` |
| `outcome.detail` | `reason` |
| `agent.who.display` | `provider/model` (or `"debug_runner"` for `answer_id IS NULL`) |
| `source.observer` | `"workbench-server"` |
| `entity[*].what` | The resources returned (resource_ids_json) |

### `evidence_claim` â†’ `Provenance.entity`

Each `evidence_claim` row corresponds to one `Provenance.entity`
referencing a FHIR resource that supported the claim. Multiple
references on a single claim become multiple `entity` entries with
the same `agent.role`.

## What the audit log is **not**

- Not a place auth tokens go. The `connection.authToken` column is
  the only place a token lives; the FHIR proxy attaches it to
  upstream requests; tools never receive it; the audit log inherits
  that. Regression test: `routes/sessions.test.ts` "never returns
  the connection's auth token in any envelope".
- Not encrypted at rest. Phase A is synthetic-only single-user
  local; see [`limitations.md`](./limitations.md).
- Not a write-back surface. Phase A is read-only against upstream
  FHIR. The mapping above is for vocabulary, not for shipping.
- Not multi-user scoped. There is no `actor_id` column; the
  workbench is a local-first single-user research tool.
